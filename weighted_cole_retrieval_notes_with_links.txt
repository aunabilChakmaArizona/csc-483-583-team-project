Scope

This note describes the retrieval path used by `python -m src.run_test_100 --weighted-cole`, including:
- every configured weight, including the zero-weight components
- what each corresponding index or component stores
- what each component searches
- how the query text is formed and processed
- how the indexes were built starting from raw wiki articles
- how scores are normalized and merged
- a few important implementation details that are easy to miss

4. Final Score Formula

The weighted score is a straight weighted sum of normalized component scores in [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:1004).

General formula used by `--weighted-cole`:

`final_score(doc) =`
`title_weight * title_score`
`+ redirect_weight * redirect_score`
`+ body_weight * body_score`
`+ first_sentence_weight * first_sentence_score`
`+ first_two_sentences_weight * first_two_sentences_score`
`+ first_paragraph_weight * first_paragraph_score`
`+ year_match_weight * year_match_score`
`+ quote_match_weight * quote_match_score`
`+ category_first_sentence_weight * category_first_sentence_score`
`+ category_first_two_sentences_weight * category_first_two_sentences_score`
`+ faiss_weight * faiss_score`
`+ natural_questions_avg_faiss_weight * natural_questions_avg_faiss_score`
`+ natural_questions_concat_faiss_weight * natural_questions_concat_faiss_score`

With the current configured `WEIGHTED_*` values, the effective formula is:

`final_score(doc) =`
`20 * body_score`
`+ 1 * faiss_score`
`+ 1 * natural_questions_avg_faiss_score`
`+ 1 * natural_questions_concat_faiss_score`
`+ 1 * year_match_score`

because all other weights are currently `0.0` in [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:136).


5. Normalization Details

- Sparse Whoosh-style component results use max normalization:
  `normalized = raw_score / max_raw_score`
  Source: [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:283)

- Dense FAISS results use min-max normalization:
  `normalized = (raw_score - min_raw_score) / (max_raw_score - min_raw_score)`
  Source: [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:301)

- Default FAISS merges the 4 document FAISS variants by taking the maximum normalized score a document receives across those variants in [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:487).

- Natural-question average FAISS averages the 5 natural-question result sets in [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:500).


6. Query Input And Processing

- The base query text is the clue alone, or `category + " " + clue` when `--include-category` is enabled in [run_test_100.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/run_test_100.py:367).

- Category-semantic components do not use the full transformed query string. They use `question.category` only, passed separately as `query_category` in [run_test_100.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/run_test_100.py:775) and consumed in [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:879).

8. Additional Details That Matter

- `WEIGHTED_COMPONENT_LIMIT = 10000`, so each component can retrieve and normalize up to 10,000 candidates before the final top-k cut in [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:165).

- Result aggregation is by canonical `(source_file, article_index)` key in [search.py](/Users/aunabilchakma/Study/PhD/csc-483-583-team-project/src/search.py:733).
